# モデル比較

> このドキュメントは Phase 3 で詳細化する。現時点では調査要件と評価基準を定義する。

## 調査対象

### 必須調査

- **Kronos** (https://github.com/shiyu-coder/Kronos) — 時系列予測モデル

### 比較候補

- LightGBM
- XGBoost
- 時系列 Transformer 系モデル
- ロジスティック回帰 / ランダムフォレスト
- ルールベース + ML フィルター

## Kronos 調査項目

| 項目 | 調査内容 |
|------|---------|
| ライセンス | 商用利用可否 |
| 依存関係 | PyTorch バージョン、CUDA 互換性 |
| モデルサイズ | パラメータ数、ファイルサイズ |
| 必要 VRAM | 16GB（RTX 4070 Ti SUPER）で動作するか |
| 入力データ形式 | OHLCV を直接受けられるか、前処理要件 |
| OHLCV への適用方法 | 時間足、窓サイズ、正規化方法 |
| 推論速度 | 1hour サイクルに間に合うか（< 30秒） |
| ファインチューニング | 必要性、コスト、データ要件 |
| 暗号資産短期売買への適用可能性 | 実績、先行研究 |
| データリークのリスク | 学習データに未来情報が含まれないか |
| 再現性 | 同じ入力で同じ出力か |
| メンテナンス状況 | 最終コミット日、Issue 対応 |
| 予測値→売買シグナル変換 | confidence score への変換方法 |
| 不確実性の扱い | 予測分布、信頼区間の出力 |

## モデル統合方針

- モデル出力 → `confidence_score` (0.0-1.0) に正規化
- `confidence_score` < 0.6 → シグナルとして使わない
- 既存ルールベース戦略のフィルターとして使用（モデル単独での売買判断はしない）
- 最終判断はリスク管理レイヤーが行う

## 採用基準

- ベースライン戦略より手数料・スリッページ込み成績が良い
- 過剰最適化の疑いが小さい
- Paper trade でも大きく崩れない
- 推論コストが現実的（< 30秒/サイクル）
- 実装と保守が可能
- 小資金運用に適している

## 不採用基準

- バックテストでしか勝てない
- 売買回数が多すぎて手数料負けする
- スプレッドに弱い
- 推論が重すぎる（VRAM 16GB に収まらない、推論 > 30秒）
- 再現性が低い
- 依存関係が危険または不明
- ライセンス上の懸念がある

## GPU/CPU 判定

```python
def get_device() -> str:
    if torch.cuda.is_available():
        vram = torch.cuda.get_device_properties(0).total_memory / (1024**3)
        if vram >= 12:  # Kronos 推定要件
            return "cuda"
    return "cpu"  # fallback
```

## 実験管理

- `src/models/experiments/` にパラメータ・結果を YAML で記録
- git tag でモデルバージョンを管理
- 重量級の MLflow / W&B は導入しない（小規模プロジェクトには過剰）

## モデル劣化検知

- 直近N日の予測精度が学習時から X% 以上低下 → アラート
- 検知時はモデルシグナルを無効化し、ルールベースのみで運用
- 再学習は手動トリガー
