# claude-mem memory on/off 比較用 golden task set

作成日: 2026-04-17
参照: docs/ai_agent_tool_adoption_review_integrated_for_claude_code.md

---

## 目的

`claude-mem` の memory 有効/無効を切り替えて同じタスクを実行し、以下を測定する。

- 過去文脈の再説明時間の差
- 同じファイル調査の繰り返し回数の差
- 回答品質の差（正確性・関連性・無関係変更の有無）
- memory 注入が原因の誤判断の有無

---

## 実行手順

1. 各タスクを「memory OFF」で実行し、所要時間・往復回数・質問数を記録する
2. 同じタスクを「memory ON」で実行し、同様に記録する
3. 差分を比較し、改善・劣化の両方を記録する
4. 結果を本ファイルの「計測結果」セクションに記入する

---

## タスク一覧

### T01: 設計判断の確認

**タスク内容:**
「このプロジェクトで bitbank を選んだ理由と、nonce 再生成が必要な理由を説明してください。」

**期待するメモリ活用:**
- 取引所選定の判断経緯（CLAUDE.md または past session）
- HMAC-SHA256 署名の実装詳細

**計測ポイント:**
- memory OFF: 設計文書 / コードを探して説明するまでの往復数
- memory ON: 過去セッションの記憶から即答できるか

**合否判定:**
- PASS: memory ON で往復数が 2 回以上減少、かつ回答の正確性が同等以上
- FAIL: 誤った nonce 再生成の理由を記憶から引き出す

---

### T02: バグ修正の文脈確認

**タスク内容:**
「`TIMEOUT_CANCELLED` で LiveState が更新されない問題の修正内容を確認し、同様のパターンがないか調べてください。」

**期待するメモリ活用:**
- progress.md に記録された Important A の詳細
- 修正対象ファイル・行番号の記憶

**計測ポイント:**
- memory OFF: progress.md と engine.py を読んで把握するまでの時間
- memory ON: 過去セッションの記憶から修正箇所を特定できるか

**合否判定:**
- PASS: memory ON で正しいファイル・行番号を最初から提示
- FAIL: 誤ったファイルや無関係な箇所を調査する

---

### T03: 実装パターンの再利用

**タスク内容:**
「`fail-closed` パターンをこのプロジェクトではどのように実装しているか確認し、新しい箇所に適用してください。対象: `_bar_loop()` 内の新しい外部 API 呼び出し（仮）。」

**期待するメモリ活用:**
- LiveGateError の使い方
- fail-closed の実装例（NF2, NF3, NF6 の修正パターン）

**計測ポイント:**
- memory OFF: fail-closed パターンをコード全体から探す時間
- memory ON: 既知パターンをすぐに適用できるか

**合否判定:**
- PASS: memory ON で既存パターンと一致した実装を提示
- FAIL: 異なる（過去に失敗した）パターンを適用する

---

### T04: テスト設計の文脈継続

**タスク内容:**
「`TestSyncInitialBalance` クラスに追加したテストの設計思想を説明し、同じ思想でもう 1 件テストを追加してください。」

**期待するメモリ活用:**
- NF2 の fail-closed 条件（BTC 残高 / ポジション不一致）
- 既存テストのパターン

**計測ポイント:**
- memory OFF: テストコードを読んで設計意図を理解するまでの時間
- memory ON: 過去セッションの設計意図から直接追加できるか

**合否判定:**
- PASS: memory ON で既存テストと整合する新テストを 1 往復以内で生成
- FAIL: 既存テストと重複または矛盾するテストを生成する

---

### T05: 複数日にまたがる実装の文脈復元

**タスク内容:**
「前回セッションで残った 'Important 2件' の内容を説明し、対応の準備をしてください。コードを読まずに記憶から答えてください。」

**期待するメモリ活用:**
- progress.md 記録: TIMEOUT_CANCELLED LiveState 更新漏れ（Important A）
- progress.md 記録: exchange_order_id 空文字列検証漏れ（Important B）

**計測ポイント:**
- memory OFF: 必ず progress.md を読む必要がある
- memory ON: 記憶から即座に 2 件を提示できるか

**合否判定:**
- PASS: memory ON で 2 件の内容・ファイル・行番号を正確に提示
- FAIL: 内容が不正確、または他の issue を混同する

## 判定基準（Phase 1 完了条件）

- 5 タスク中 4 タスク以上で memory ON が往復回数 1 回以上削減
- memory 注入による明確な誤判断が 0 件
- 誤想起が発生した場合、検知・訂正できた件数を記録する
