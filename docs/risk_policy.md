# リスク管理方針

## 前提

- 初期資金: 100,000 JPY
- 取引所: bitbank
- 銘柄: BTC/JPY 現物
- 最大同時ポジション: 1

## 資金管理

| 項目 | 値 | 根拠 |
|------|-----|------|
| Paper trade 初期資金 | 100,000 JPY | 実運用と同額でシミュレーション |
| Live 移行時 初期投入額 | 30,000 JPY | 全額でなく段階的に投入。残り7万円は検証後に追加 |
| 1取引あたり最大リスク | 現在残高の 2% | ケリー基準の保守的適用 |
| 最大ポジションサイズ | 現在残高の 30% | 集中リスク回避 |
| 最大同時ポジション数 | 1 | MVP は単一銘柄・単一ポジション |

### 動的リスク計算の原則

**全ての損失制限は固定JPY額ではなく、サイクル開始時の現在残高に対するパーセンテージで動的計算する。**

```python
# サイクル開始時に残高を取得してから閾値を計算する
current_balance = portfolio.get_current_balance()
max_trade_loss   = current_balance * 0.02   # 1取引最大損失 2%
daily_loss_limit = current_balance * 0.03   # 日次最大損失 3%
weekly_loss_limit = current_balance * 0.05  # 週次最大損失 5%
monthly_loss_limit = current_balance * 0.10 # 月次最大損失 10%
max_drawdown_limit = peak_balance * 0.15    # 最大DD はピーク残高から計算
```

設計書内の「JPY 額例」（2,000 JPY 等）は初期資金 100,000 JPY 時点の参考値。
コードでは定数化せず、常に現在残高から計算すること。

## 損失制限（Circuit Breaker）

| 制限 | 閾値（現在残高に対する%） | 動作 |
|------|------|------|
| 1取引最大損失 | **現在残高の 2%** | 即座にポジションクローズ |
| 日次最大損失 | **現在残高の 3%** | 当日の新規注文を全停止 |
| 週次最大損失 | **現在残高の 5%** | 週末まで全停止 |
| 月次最大損失 | **現在残高の 10%** | 月末まで全停止、手動レビュー必須 |
| 最大ドローダウン | **ピーク残高から 15%** | **Kill switch 発動、自動再開しない** |
| 連敗停止 | 5連敗（ラウンドトリップ） | 全停止、手動レビュー必須 |

### 連敗カウントの定義

「1トレード」= エントリー（BUY）からエグジット（SELL）までのラウンドトリップ1回。
注文単位ではなくポジション単位で計測する。損失決済が連続5回で停止する。
- ✓ カウント対象: エグジット時点で損益がマイナスのラウンドトリップ
- ✗ 対象外: ノートレード判断、未約定注文のキャンセル、部分約定のみの注文

### 停止後の動作

- 日次停止: 翌日 0:00 JST に自動解除
- 週次停止: 翌週月曜 0:00 JST に自動解除
- 月次停止: **自動解除しない**。手動レビュー後に config 変更で解除
- 最大ドローダウン停止: **自動解除しない**。Kill switch 発動

## 市場環境による停止条件

| 条件 | 閾値 | 根拠 |
|------|------|------|
| スプレッド拡大 | Bid/Ask スプレッド > 0.5% | スリッページリスクが手数料を超える |
| 流動性不足 | Best bid/ask の板厚 < 0.01 BTC | 成行注文のインパクトが大きすぎる |
| 異常ボラティリティ | 直近1時間の変動率 > 5% | 急変相場での損失リスク |
| API エラー | 連続3回の API エラー | 取引所障害の可能性 |
| API レスポンス遅延 | レスポンス > 5秒 | 約定タイミングのずれ |

市場環境停止はサイクル単位。次のサイクルで条件が解消されていれば自動的に取引再開。

## テールリスク指標（CVaR）

Circuit Breaker の閾値設定に加え、テールリスクを Conditional Value at Risk (CVaR) で補完する。
Sharpe/Sortino は正規分布を前提とするため、BTC の重い裾を過小評価する。

| 指標 | 定義 | 目安 |
|------|------|------|
| **CVaR(95%)** | 損失が大きい下位5%のトレードにおける平均損失率 | < -3% / トレード |
| 最大単一損失 | 1トレードの最大観測損失 | < -5% |

### CVaR の運用ルール

| 項目 | 値 |
|------|-----|
| 計算ウィンドウ | 直近 50 トレード（ラウンドトリップ）のリターン系列 |
| 更新頻度 | 毎日 0:00 JST（日次バッチ）および新規エグジット発生時 |
| 最低サンプル数 | 20 トレード未満の場合は CVaR を計算しない（Not Available と記録） |

### CVaR 閾値超過時の動作

| CVaR(95%) | 動作 |
|-----------|------|
| -3% ～ -4% | 警告ログ (`CVaR_WARNING`) を記録。取引継続 |
| -4% ～ -5% | Signal confidence を 0.5 倍にダウングレード。ポジションサイズを半減 |
| < -5% | 戦略を停止（`CVAR_BREACH_STOP`）。手動レビューまで再開しない |

CVaR が基準を超えた戦略は、Circuit Breaker が発動するより前に
シグナル信頼度を下げるか戦略を停止する。

## Kill Switch

### 発動条件

1. 最大ドローダウン到達（15%）
2. 月次損失上限到達（10%）
3. 手動トリガー（CLI / config 変更）
4. 取引所 API 長時間障害（10分以上）

### 発動時の動作

1. 全未約定注文をキャンセル（paper: 記録のみ）
2. 全ポジションをクローズ（paper: 記録のみ）
3. 新規注文を禁止するフラグを DB に記録
4. ログに `KILL_SWITCH_ACTIVATED` を記録
5. レポート生成

**Live 版の注意**: Kill switch 発動時のポジションクローズは成行注文になる。
市場急変（HIGH_VOL レジーム）での Kill switch 発動は最悪のスリッページ環境で
執行される可能性がある。0.05% の固定スリッページ仮定は楽観的すぎる場合がある。
Kill switch 時の追加スリッページ（0.2〜0.5%）を想定したコスト感度分析を行うこと。

### Kill Switch 失敗後の最終状態（R-C2）

ポジションクローズ（成行注文）がリトライ3回後も失敗した場合:

1. システムを **`MANUAL_INTERVENTION_REQUIRED`** 状態に遷移させる
2. 全処理を即時停止（新規注文・シグナル生成・データ取得ループを停止）
3. 以下の内容を含む緊急アラートをログ・標準エラー出力に記録する:
   ```
   [CRITICAL] KILL_SWITCH_CLOSE_FAILED
   - 未クローズポジション: {pair}, {size} BTC, 取得単価 {entry_price} JPY
   - 取引所 Web UI で手動クローズが必要です
   - URL: https://bitbank.cc/
   - 最終エラー: {last_error}
   ```
4. `MANUAL_INTERVENTION_REQUIRED` 状態のシステムは、手動で `--reset-intervention` フラグを渡した起動でのみリセット可能。自動リセット不可。

### 再開条件

Kill Switch 解除は **設定ファイル変更ではなく、環境変数＋CLI による二段階認証** で行う:

```bash
# Step 1: 環境変数でセッション単位の許可を宣言
export KILL_SWITCH_OVERRIDE=1

# Step 2: CLI の対話確認を経て起動（--override-kill-switch フラグ必須）
python -m crypt.main --override-kill-switch
# > Kill switch が発動中です。停止理由: {reason}
# > 再開しますか？ [yes/NO]: yes   ← 明示的な入力が必要
```

- `config.yaml` の値だけでは Kill Switch を解除できない設計にする
- 環境変数 `KILL_SWITCH_OVERRIDE` はプロセス起動時のみ参照し、実行中は再読み込みしない
- 再開時に停止理由のレビューログを必ず記録する
- **自動再開は一切行わない**

## Paper Trade でのリスク制限検証

以下が正しく動作することを paper trade で確認してから live 移行を検討する:

1. [ ] 日次損失上限で停止すること
2. [ ] 連敗停止が発動すること
3. [ ] スプレッド拡大時にノートレードになること
4. [ ] Kill switch が発動すること
5. [ ] Kill switch 発動後、自動再開しないこと
6. [ ] 全判断のログが残ること
7. [ ] 停止後に手動でのみ再開できること

## Live 移行条件

以下を**全て**満たすまで live 取引しない:

- [ ] バックテストで手数料・スリッページ込み期待値がプラス
- [ ] Buy & Hold よりリスク調整後成績が良い
- [ ] ウォークフォワード検証で大きく崩れていない
- [ ] Paper trade を最低2週間以上実施
- [ ] Paper trade で想定外の注文・残高不整合がない
- [ ] Kill switch が動作する
- [ ] 日次損失上限が動作する
- [ ] 最大ポジション制限が動作する
- [ ] API エラー時に停止する
- [ ] 全判断ログが保存される
- [ ] API キーに出金権限がない
- [ ] 手動停止手順が README にある
- [ ] ユーザーが明示的に live 移行を許可する

## 手動停止手順

1. `config.yaml` の `mode` を `paper` に変更
2. プロセスを停止（`Ctrl+C` or `kill`）
3. 未約定注文があれば手動で取引所 UI から確認・キャンセル
4. `logs/` で最終状態を確認
